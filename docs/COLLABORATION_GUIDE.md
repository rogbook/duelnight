# DuelNight 협업 개발 가이드 — Lovable + Antigravity + Claude

> **최종 수정**: 2026-06-03
> **적용 대상**: 프로젝트에 참여하는 모든 도구와 개발자
>
> 이 문서는 **Lovable**(UI/프로토타이핑) · **Antigravity**(로직/리팩토링) · **Claude Cowork**(분석/버그수정/문서화)
> 세 도구가 충돌 없이 협력하기 위한 표준 워크플로우를 정의합니다.

## 1. 각 도구의 역할 정의

| 도구 | 주요 역할 | 하지 말아야 할 것 |
|------|----------|-----------------|
| **Lovable** | UI 컴포넌트 생성, Supabase 스키마/RLS/마이그레이션, 패키지 추가, 배포(Publish) | 복잡한 비즈니스 로직 직접 구현, 서버 함수 대규모 수정 |
| **Antigravity** | 복잡한 알고리즘 구현, 대규모 리팩토링, 로컬 디버깅, 단위 테스트 작성 | Lovable 관리 파일 수정 (아래 목록 참고), 무단 패키지 추가 |
| **Claude Cowork** | 버그 분석·수정, 코드 리뷰, 보안 취약점 탐지, 문서 작성·갱신 | DB 스키마 직접 변경, 배포(Publish) 직접 실행 |

### 각 도구별 강점 요약

```
Lovable      → 빠른 UI 생성 + Supabase 자동 연동
Antigravity  → 에디터 수준의 정밀 코딩 + 멀티파일 리팩토링
Claude       → 전체 코드베이스 분석 + 버그 탐지 + 문서 자동화
```

```

### STAGE 1 — Lovable: 기초 개발 및 UI 프로토타이핑

> 새 기능의 시작점. 빠른 UI 구현과 DB 구조 정의.

- 새 페이지·컴포넌트 생성 (React + Tailwind)
- Supabase 테이블 · RLS 정책 · Edge Function 관리
- npm/bun 패키지 추가 (lock 파일 무결성 유지)
- 디자인 토큰 (`styles.css`, Tailwind config) 정의
- 작업 완료 후 **GitHub에 자동 Push** → 다른 도구가 pull 가능

### STAGE 2 — Antigravity: 정밀 개발 및 로직 구현

> Lovable이 만든 UI 위에 복잡한 비즈니스 로직 추가.

**시작 전 필수 체크:**
```bash
git checkout main
git pull origin main        # Lovable 변경분 반드시 동기화
git checkout -b feature/<작업명>
```

**주요 작업:**
- 복잡한 알고리즘 구현 (통계 계산, 데이터 처리 등)
- 대규모 리팩토링 및 단위 테스트 작성
- 로컬 개발 환경에서 정밀 디버깅
- 서버 함수(TanStack Server Functions) 고도화

**완료 후:**
```bash
git add .
git commit -m "feat: 작업 내용 요약"
git push origin feature/<작업명>
# GitHub에서 PR 생성 → main 머지
```

### STAGE 3 — Claude Cowork: 분석·수정·문서화

> 코드베이스 전반 분석, 버그 탐지 및 수정, 문서 최신화.

**투입 시점:**
- 기능 구현 후 배포 전 버그 검토 요청 시
- 보안/성능 이슈 발생 시
- 문서 갱신이 필요할 때

**주요 작업:**
- 전체 코드 정적 분석 → 버그 리스트 도출
- Critical/High 버그 즉시 수정 (직접 파일 편집)
- `docs/BUGFIX_LOG.md` 수정 이력 기록
- `docs/RELEASES.md` 릴리스 노트 갱신
- 협업 문서(`COLLABORATION_GUIDE.md` 등) 최신화

**완료 후:**
- 수정된 파일 목록과 이유를 `docs/BUGFIX_LOG.md`에 기록
- 로컬에서 `git pull` → `git push`로 GitHub 반영

### STAGE 4 — Lovable: 최종 검증 및 배포

> QA 통과 후 실운영 배포.

- Preview URL에서 전체 기능 QA (`docs/QA_CHECKLIST.md` 기준)
- UI 미세 조정 (Visual Edits)
- 이상 없으면 **Publish** 버튼으로 실운영 반영
- `docs/RELEASES.md`에 릴리스 노트 추가

## 3. 충돌 방지 핵심 원칙

### 황금률: "한 번에 한 곳에서만 같은 파일을 수정"

- **동시 편집 금지**: 같은 파일을 두 도구가 동시에 수정하면 충돌 발생
- **전환 전 반드시 push**: 도구를 바꾸기 전에 작업 내용을 GitHub에 반영
- **시작 전 반드시 pull**: 어떤 도구든 작업 시작 전 최신 코드 동기화

### 도구 전환 체크리스트

```
Lovable → Antigravity:  Lovable에서 모든 변경이 GitHub에 Sync됐는지 확인 후 git pull
Lovable → Claude:       위와 동일
Antigravity → Lovable:  git push 완료 확인 + PR 머지 후 전환
Antigravity → Claude:   git push 완료 확인
Claude → Lovable:       Claude 수정사항 commit+push 완료 확인
Claude → Antigravity:   위와 동일
```

### 충돌 발생 시 우선순위

```
자동생성 파일 (types.ts, routeTree.gen.ts 등)  → Lovable 버전 우선
비즈니스 로직 파일                              → 최신 커밋 기준 수동 머지
문서 파일 (docs/*.md)                          → 두 버전 내용 합산 후 수동 머지
```

## 4. Lovable 관리 파일 — 수정 금지 목록

아래 파일은 Lovable이 자동 생성/관리합니다. Antigravity와 Claude는 **절대 수정하지 마세요.**
수동으로 고치면 Lovable이 다음 Sync/Publish 때 덮어쓰거나 충돌이 나서 **배포가 깨질 수 있습니다.**

> 💡 파일 상단에 `// This file is automatically generated. Do not edit it directly.` 주석이 있으면 무조건 Lovable 관리 파일입니다.

| 파일 | 이유 |
|------|------|
| `src/integrations/supabase/types.ts` | Supabase 스키마 타입 자동 생성 |
| `src/integrations/supabase/client.ts` | 브라우저용 Cloud 인증 클라이언트 |
| `src/integrations/supabase/client.server.ts` | 서버용 service-role 클라이언트 (RLS 우회, 자동 생성) |
| `src/integrations/supabase/auth-attacher.ts` | serverFn 호출에 Bearer 토큰 부착 (자동 생성) |
| `src/integrations/supabase/auth-middleware.ts` | SSR 요청에 사용자 세션 주입 (자동 생성) |
| `src/routeTree.gen.ts` | TanStack Router 자동 생성 |
| `.env` (`.env.*` 포함) | Lovable Secrets 관리 |
| `package-lock.json` / `bun.lock` | 패키지 lock 무결성 |
| `supabase/config.toml` (project_id) | 프로젝트 ID 고정값 |

> **원칙**: `src/integrations/supabase/` 디렉토리 **전체**를 Lovable 전용으로 취급하세요.
> 새 파일이 추가돼도 자동 생성 주석이 있으면 손대지 않습니다. Supabase 스키마·RLS·클라이언트 변경이 필요하면 반드시 **Lovable에서** 작업한 뒤 GitHub로 sync 합니다.

> GitHub Actions `.github/workflows/guard-lovable-files.yml`이 PR에서 위 파일 변경을 감지하면 자동으로 경고 코멘트를 남기고 체크를 실패시킵니다.
>
> ⚠️ **현재 guard 워크플로우는 일부 파일을 누락하고 있습니다** (2026-06-03 점검):
> - lock 파일 정규식이 실제 파일명 `bun.lock`이 아닌 `bun.lockb`를 검사 → **`bun.lock`이 보호되지 않음**
> - `client.server.ts` / `auth-attacher.ts` / `auth-middleware.ts` / `routeTree.gen.ts` 미포함
>
> 위 파일을 실수로 커밋해도 CI가 잡지 못하므로, guard 정규식을 아래와 같이 갱신하는 것을 권장합니다:
> ```
> ^(\.env(\..+)?|src/integrations/supabase/(client(\.server)?|types|auth-attacher|auth-middleware)\.ts|src/routeTree\.gen\.ts|package-lock\.json|bun\.lock|yarn\.lock|pnpm-lock\.yaml)$
> ```
> guard가 갱신되기 전까지는 **수동 규율(이 목록)** 이 유일한 방어선입니다.

## 4-1. 퍼블리싱 필수 불변식 (깨지면 배포 실패)

아래는 Antigravity/Claude 작업 중 **절대 건드리거나 제거하면 안 되는** 런타임 연결고리입니다.
Lovable Publish 후 인증/서버 함수가 동작하려면 반드시 유지되어야 합니다.

| 불변식 | 위치 | 깨질 때 증상 |
|--------|------|-------------|
| `functionMiddleware: [attachSupabaseAuth]` 등록 | `src/start.ts` | serverFn RPC에 Bearer 토큰 미부착 → 모든 인증 의존 서버 호출이 401/빈 결과 |
| `AuthProvider`의 `onAuthChange` → `router.invalidate()` + `queryClient.invalidateQueries()` | `src/routes/__root.tsx` | 로그인/로그아웃 후 화면·캐시 미동기화 (로그인 풀림 플래시) |
| `getSession()` 복원 결과를 첫 인증 기준으로 사용 | `src/hooks/use-auth.tsx` | HMR/리로드 직후 세션이 `null`로 잠깐 잡혀 로그아웃처럼 보임 |
| 인증 의존 쿼리는 `authLoading` 종료 후 실행 | `use-is-admin.ts`, `stores.index.tsx`, `stores.$id.tsx` | 세션 복원 전 쿼리 실행 → 불안정한 결과 |

> 이 4건은 2026-06-02 Lovable 핫픽스(`90b6196`)로 정착된 패턴입니다. 리팩토링 시 동작을 보존하세요. 자세한 내용은 [`BUGFIX_LOG.md`](./BUGFIX_LOG.md) 2026-06-02 항목 참고.

## 5-1. 도메인별 작업 분담 규칙 (지시 전 결정 기준)

> 코드 수정을 어느 도구에 지시할지 헷갈릴 때 이 규칙을 따르세요.
> **핵심**: 이 프로젝트의 "백엔드"는 두 종류입니다.
> ① **Supabase 백엔드**(DB·RLS·마이그레이션·Edge Function·`types.ts`) → 자동 생성 연동이라 **Lovable 전용**
> ② **앱 서버 코드**(`*.server.ts`, `*.functions.ts` = TanStack 서버 함수) → 일반 코드라 **Antigravity/Claude 작업 가능**

### 판단 3단계 (지시 전에 이 순서로 자문)

```
Q1. Lovable 관리 영역을 건드리나?
    (DB스키마·RLS·마이그레이션 / Edge Function / 패키지 추가 /
     .env·Secrets / 새 UI·디자인 / 자동생성 파일 — 섹션 4 참고)
        → 예  →  ★ Lovable에서 지시

Q2. (아니오) 새로 만드는 복잡한 로직 / 멀티파일 리팩토링 /
            로컬 테스트 필요 / 보안 민감(결제 검증 등)?
        → 예  →  ★ Antigravity에서 지시

Q3. (아니오) 버그 수정 / 코드리뷰 / 보안점검 / 문서 / 작은 외과적 수정?
        → 예  →  ★ Claude에서 지시
```

> 한 줄 요약: **새로 만들면 Lovable(UI)·Antigravity(로직), 고치고 검토하면 Claude.**

### 도메인별 담당 (실제 파일 기준)

| 영역 | 작업 | 담당 | 비고 |
|------|------|------|------|
| **프론트** | 새 페이지·컴포넌트·디자인 | **Lovable** | UI 자동화 강점 |
| | 기존 UI 버그·작은 수정 | **Claude** | 외과적 수정 |
| | 복잡한 상태·멀티파일 UI 리팩토링 | **Antigravity** | IDE 정밀편집 |
| | 시각 미세조정 | **Lovable** | Visual Edits |
| **백엔드 (Supabase)** | DB 스키마·RLS·마이그레이션·Edge Function | **Lovable** | ⚠️ `types.ts` 자동동기화 때문에 **반드시 Lovable** |
| | 마이그레이션 SQL 사전 검증 | Antigravity(검증) → **Lovable**(적용) | 적용은 Lovable만 |
| **백엔드 (앱 서버코드)** | `*.functions.ts`·`*.server.ts` 새 로직 | **Antigravity** | `client.server.ts`만 예외(손대지 말 것) |
| | 위 파일 버그·보안 수정 | **Claude** | |
| **결제/외부 API** | PG 키·Secrets | **Lovable** | LIVE 키 |
| | 결제 UI (`PaymentDialog.tsx`, `store.tsx`) | **Lovable**(표시) / **Antigravity**(연동로직) | |
| | 클라이언트 SDK (`payment.ts`) | **Antigravity** | PortOne/PayPal 연동 |
| | **서버 검증** (`payment.functions.ts`) | **Antigravity** | 보안민감 + 로컬테스트 |
| | Webhook용 `verify-payment` Edge Function | **Lovable** | Supabase Edge Function |
| | 연동 후 보안 리뷰 | **Claude** | 결제는 **반드시** 리뷰 |

### 결제 연동 권장 시나리오 (도구가 섞이므로 순서 준수)

```
1. Lovable     → 결제 UI/모달 화면 + payments 테이블·RLS 생성
2. Antigravity → payment.ts SDK 연동 + payment.functions.ts 서버 검증 로직
                 (로컬에서 테스트 모드로 결제 왕복 검증)
3. Lovable     → verify-payment Edge Function 배포 + 웹훅 URL 등록 + LIVE 키 Secrets
4. Claude      → 서버 검증/웹훅 보안 리뷰 (금액 위변조·중복결제·권한)
5. Lovable     → QA 통과 후 Publish
```

### 지시할 때 한 문장 템플릿

- **Lovable**: "○○ 화면 만들어줘 / payments 테이블에 ○○ 컬럼 추가하고 RLS 걸어줘"
- **Antigravity**: "feature 브랜치 따서 payment.functions.ts에 결제 검증 로직 구현하고 로컬 테스트까지 해줘"
- **Claude**: "결제 서버 코드 보안 취약점 점검하고 BUGFIX_LOG에 기록해줘"

> 이 규칙대로 지시하면 **① 같은 파일 동시 편집 ② Lovable 자동생성물 충돌 ③ 결제 보안 누락**
> 세 가지가 구조적으로 차단되어 퍼블리싱 사고를 예방합니다.

## 7. 환경변수 관리

| 변수 위치 | 관리 주체 | 용도 |
|-----------|----------|------|
| Lovable Secrets | **Lovable** | 실운영 환경변수 (SUPABASE_URL 등) |
| `.env.local` | **Antigravity** | 로컬 개발 전용 변수 |
| `.env` | **Lovable (읽기 전용)** | 공유 환경변수 — Antigravity·Claude는 수정 금지 |

### 퍼블리싱에 필요한 서버 측 Secrets (필수)

서버 함수의 admin 클라이언트(`client.server.ts`)와 SSR 인증이 동작하려면 **Lovable Secrets에 아래 값이 반드시 설정**되어 있어야 합니다. 누락 시 Publish 후 서버 함수가 런타임에서 throw 합니다.

| 변수명 | 사용처 | 누락 시 증상 |
|--------|--------|-------------|
| `SUPABASE_URL` | `client.server.ts` | `Missing Supabase environment variable(s)` throw |
| `SUPABASE_SERVICE_ROLE_KEY` | `client.server.ts` (RLS 우회 admin 작업) | 위와 동일 — 관리자/서버 함수 전체 실패 |

> ⚠️ `SUPABASE_SERVICE_ROLE_KEY`는 **service-role 키(RLS 우회)** 이므로 절대 클라이언트 코드/브라우저로 노출하면 안 됩니다. 서버 함수·서버 라우트에서만 `supabaseAdmin`을 통해 사용하세요.

> 새 환경변수 추가 시 반드시 `docs/BUGFIX_LOG.md` 또는 `docs/RELEASES.md`의 `⚙️ ENV 필요` 섹션에 명시하여 다른 도구가 누락 없이 설정할 수 있게 합니다.

## 8. 문제 발생 시 복구

| 상황 | 조치 |
|------|------|
| 빌드 실패 / 캐시 오류 | Lovable 프리뷰 강제 새로고침 (Ctrl+Shift+R) 또는 빌드 재실행 |
| 로컬 작업 유실 | `git reflog`로 시점 복구 |
| 잘못된 코드가 main에 머지됨 | `git revert <커밋해시>` 후 push → Lovable에서 재배포 |
| DB 마이그레이션 실수 | `docs/DEPLOY_PROCESS.md` 롤백 절차 참고 |
| Claude 수정 후 빌드 오류 | `docs/BUGFIX_LOG.md` 최신 항목 확인 → Antigravity에서 로컬 검증 후 재수정 |
