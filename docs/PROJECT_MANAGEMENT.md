# DuelNight 프로젝트 관리 가이드

작성일: 2026-06-09
문서 목적: 개발을 잘 모르는 운영자도 **무엇을 먼저 해야 하는지**, **진행 상황을 어떻게 확인해야 하는지**, **배포 전에 무엇을 검증해야 하는지** 한눈에 볼 수 있도록 프로젝트 관리 기준을 정리합니다.

---

## 현재 운영 방향

2026-06-09 기준으로 **Lovable에서는 신규 작업을 진행하지 않고, 기존 협업 도구 분담 방식도 중단합니다.** 앞으로의 개발·배포·DB 변경은 `docs/CODEX_WORKFLOW.md`의 Codex 중심 흐름과 GitHub 기준으로 관리합니다. 기존 Lovable 산출물은 `docs/LOVABLE_HANDOFF.md`에 정리합니다.

---

## 1. 운영 원칙

DuelNight는 기능을 많이 만드는 것보다 **사용자가 안전하게 쓸 수 있는 상태로 작게 자주 개선**하는 것을 우선합니다.

1. **한 번에 하나의 목표만 진행**합니다.
   - 예: “카드 검색 개선”과 “결제 개선”을 동시에 크게 바꾸지 않습니다.
2. **데이터를 지키는 변경을 최우선**으로 봅니다.
   - 로그인, 결제, DB, 권한, 파일 업로드 변경은 일반 UI 변경보다 더 엄격하게 검토합니다.
3. **문서와 코드가 같이 움직이게 관리**합니다.
   - 기능을 바꾸면 관련 기획·QA·배포 문서도 함께 갱신합니다.
4. **Preview에서 확인한 뒤 Production에 배포**합니다.
   - 운영 데이터에 영향을 줄 수 있는 변경은 바로 운영 배포하지 않습니다.

## 2. 프로젝트 보드 추천 구조

GitHub Projects, Notion, Linear, Trello 중 어떤 도구를 써도 아래 칸만 유지하면 됩니다.

| 칸 | 의미 | 이동 기준 |
|----|------|-----------|
| **Inbox** | 아이디어, 버그 제보, 요청이 처음 들어오는 곳 | 아직 우선순위를 정하지 않음 |
| **Backlog** | 언젠가 할 일 | 목표/범위가 대략 정해짐 |
| **Ready** | 바로 개발 가능한 일 | 요구사항, 화면, 검증 기준이 정리됨 |
| **In Progress** | 지금 작업 중 | 담당자 1명이 실제 작업 중 |
| **Review / QA** | 코드 검토 또는 Preview 확인 중 | 테스트 URL이나 확인 방법이 있음 |
| **Ready to Release** | 운영 배포 대기 | 배포 전 체크리스트 통과 |
| **Done** | 배포 및 문서 정리 완료 | 운영 반영, 변경 기록 작성 완료 |

> 초보 운영자 기준으로는 **Ready에 들어간 일만 개발 요청**한다고 생각하면 안전합니다.

## 3. 우선순위 기준

작업 우선순위는 감이 아니라 아래 기준으로 정합니다.

### P0 — 즉시 대응
- 로그인 불가, 결제 오류, 데이터 유실, 개인정보 노출 가능성
- 운영 서비스가 깨지는 빌드/배포 장애
- 관리자 권한 또는 RLS 같은 보안 문제

### P1 — 이번 주 처리
- 핵심 사용자 흐름 장애
  - 카드 검색, 덱 저장, 전적 등록, 컬렉션 관리 등
- 사용자가 우회하기 어려운 모바일 UI 문제
- 운영자가 데이터 등록·검수에 막히는 문제

### P2 — 다음 릴리스 후보
- 사용성 개선
- 통계/필터/정렬 개선
- 문구, 빈 상태, 도움말 개선

### P3 — 보류 가능
- 있으면 좋은 기능
- 디자인 디테일 개선
- 장기 실험 기능

## 4. 작업 카드 템플릿

새 작업을 만들 때는 아래 양식을 복사해서 사용합니다.

```md
## 목적
이 작업으로 사용자가 무엇을 더 잘할 수 있나요?

## 범위
- 포함:
- 제외:

## 화면/경로
- 예: `/cards`, `/decks`, `/matches`

## 완료 기준
- [ ] 사용자가 해야 할 핵심 동작이 성공한다.
- [ ] 빈 데이터, 로딩, 오류 상태가 깨지지 않는다.
- [ ] 모바일 화면에서 확인했다.
- [ ] 관련 문서를 갱신했다.

## 위험도
- 낮음 / 보통 / 높음
- 높다면 이유: 로그인, 결제, DB, 권한, 파일 업로드, 외부 API 중 해당 항목 작성

## 테스트 방법
1.
2.
3.
```

## 5. 주간 운영 루틴

매주 같은 흐름으로 관리하면 누락이 줄어듭니다.

### 월요일 — 계획
- Inbox를 비우고 Backlog 또는 보류로 이동합니다.
- P0/P1을 먼저 확인합니다.
- 이번 주 목표를 1~3개만 정합니다.

### 수요일 — 중간 점검
- In Progress가 너무 많으면 새 작업을 시작하지 않습니다.
- 막힌 작업은 원인만 짧게 적고 범위를 줄입니다.

### 금요일 — 릴리스 판단
- Review / QA에 있는 작업만 배포 후보로 봅니다.
- `docs/QA_CHECKLIST.md`와 `docs/DEPLOY_PROCESS.md` 기준으로 Preview 확인 후 배포 여부를 결정합니다.
- 배포했다면 `docs/RELEASES.md`에 사용자 관점의 변경 내용을 남깁니다.

## 6. 배포 전 확인 흐름

운영 배포 전에 아래 순서를 지킵니다.

1. **개발 검증**
   - 빌드가 성공했는지 확인합니다.
   - 변경된 화면의 기본 동작을 확인합니다.
2. **Preview QA**
   - 로그인 사용자와 비로그인 사용자를 나눠 확인합니다.
   - 관리자 기능은 관리자 계정으로만 확인합니다.
3. **위험 기능 추가 확인**
   - 결제, 인증, 권한, DB, 파일 업로드, 외부 API 변경은 별도 체크를 추가합니다.
4. **문서 갱신**
   - 배포 절차: `docs/DEPLOY_PROCESS.md`
   - QA 항목: `docs/QA_CHECKLIST.md`
   - 릴리스 기록: `docs/RELEASES.md`
   - DB 작업 규칙: `docs/DB_WORKFLOW.md`

## 8. 의사결정 기록 방식

중요한 결정은 채팅에만 남기지 말고 문서에 남깁니다.

### 기록해야 하는 결정
- 결제사를 바꾸거나 추가하는 결정
- DB 구조 또는 권한 정책 변경
- 무료/유료 정책 변경
- 주요 메뉴 추가·삭제
- 운영 환경과 테스트 환경 분리

### 간단한 기록 양식

```md
## 결정일
2026-06-09

## 결정 내용
무엇을 선택했는지 한 문장으로 작성합니다.

## 이유
왜 이 선택을 했는지 2~3줄로 작성합니다.

## 대안
검토했지만 선택하지 않은 방법을 적습니다.

## 후속 작업
- [ ] 해야 할 일
```

## 9. 초보 운영자를 위한 요청 문장 예시

개발 요청은 아래처럼 쓰면 실수가 줄어듭니다.

### 좋은 요청
> `/matches` 전적 목록에서 모바일 화면일 때 승/패 배지가 너무 작습니다. 모바일에서 쉽게 보이도록 키워 주세요. 기존 기능은 바꾸지 말고 UI만 수정해 주세요. Preview에서 확인할 수 있게 해 주세요.

### 위험한 요청
> 전적 화면 전체를 더 좋게 바꿔줘.

위험한 요청은 범위가 넓어서 예상치 못한 버그가 생기기 쉽습니다. **경로, 문제, 원하는 결과, 제외 범위**를 같이 적어 주세요.

## 10. 최소 점검 체크리스트

작업을 완료 처리하기 전에 아래만큼은 확인합니다.

- [ ] 사용자가 실제로 클릭하는 주요 흐름이 성공했다.
- [ ] 빈 데이터 화면이 어색하지 않다.
- [ ] 오류가 났을 때 사용자가 이해할 수 있는 안내가 있다.
- [ ] 모바일에서 레이아웃이 깨지지 않는다.
- [ ] 관리자 전용 기능이 일반 사용자에게 노출되지 않는다.
- [ ] 변경 내용이 관련 문서 또는 릴리스 노트에 남아 있다.
