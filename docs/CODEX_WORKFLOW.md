# DuelNight Codex 단독 작업 운영 가이드

작성일: 2026-06-09
상태: **앞으로 개발 작업은 협업 도구 분담 방식이 아니라 Codex 중심으로 진행합니다.**

---

## 1. 운영 결정

DuelNight는 기존의 “Lovable + Antigravity + Claude 협업” 흐름을 중단하고, 앞으로는 **Codex가 GitHub 저장소를 기준으로 직접 작업**하는 방식으로 운영합니다.

초보자 관점에서 쉽게 말하면 다음과 같습니다.

- 예전 방식: 여러 도구가 각자 다른 역할을 맡아 작업했습니다.
- 새 방식: Codex가 저장소를 읽고, 수정하고, 검사하고, 커밋과 PR까지 한 흐름으로 처리합니다.
- 중요한 원칙: Codex도 추측으로 크게 바꾸지 않고, 작은 단위로 고친 뒤 검증합니다.

---

## 2. 이제 사용하지 않는 협업 방식

아래 방식은 **신규 작업에 사용하지 않습니다.** 기존 문서에 남아 있더라도 과거 이력으로만 봅니다.

| 기존 방식 | 앞으로의 처리 |
|-----------|---------------|
| Lovable에서 UI 생성 후 다른 도구가 보강 | Codex가 기존 코드 패턴을 보고 직접 수정 |
| Lovable에서 DB 변경 적용 | Codex가 SQL/마이그레이션 계획을 작성하고 새 DB 운영 절차로 적용 |
| Antigravity에서 로직 구현 후 Claude가 검토 | Codex가 구현과 1차 검증을 같은 작업 단위에서 수행 |
| Claude/Antigravity/Lovable 간 파일 담당 분리 | Codex가 작업 범위를 명확히 정하고 Git diff 기준으로 관리 |
| Lovable Publish로 운영 배포 | 새 배포 환경이 확정된 뒤 GitHub 기반 배포로 전환 |

---

## 3. Codex 작업 표준 흐름

Codex 작업은 항상 아래 순서로 진행합니다.

1. **요청 범위 확인**
   - 어떤 화면, 어떤 기능, 어떤 문서를 바꾸는지 먼저 정합니다.
   - 범위가 넓으면 작은 작업 카드로 나눕니다.
2. **관련 코드와 문서 읽기**
   - 기존 유틸, 타입, 컴포넌트, 문서가 있는지 먼저 확인합니다.
   - 추측으로 새 구조를 만들지 않습니다.
3. **작게 수정**
   - 한 번에 여러 핵심 기능을 바꾸지 않습니다.
   - 로그인, 결제, DB, 파일 업로드, 외부 API는 특히 작게 나눕니다.
4. **검증**
   - 가능하면 lint, build, type check, 관련 스크립트를 실행합니다.
   - 환경 문제로 실행하지 못하면 그 이유를 기록합니다.
5. **문서 갱신**
   - 운영자가 알아야 하는 변경은 `docs/`에 남깁니다.
6. **커밋과 PR 작성**
   - 변경 이유, 변경 파일, 검증 결과를 PR에 남깁니다.

## 4. Codex에 요청할 때 좋은 형식

아래 형식으로 요청하면 실수가 줄어듭니다.

```md
## 목표
무엇을 해결하고 싶은지 한 문장으로 적습니다.

## 바꿀 곳
예: `/cards`, `/matches`, `docs/DEPLOY_PROCESS.md`

## 원하는 결과
사용자 또는 운영자가 어떻게 달라져야 하는지 적습니다.

## 건드리지 말 것
이번 작업에서 제외할 범위를 적습니다.

## 확인 방법
가능하면 직접 확인할 시나리오를 적습니다.
```

### 예시

> `/matches` 모바일 화면에서 승/패 표시가 잘 안 보입니다. 전적 저장 로직은 건드리지 말고, 모바일 UI만 보기 쉽게 수정해 주세요. 수정 후 빌드 가능 여부와 변경 파일을 알려 주세요.

## 5. 작업 우선순위

Codex 작업도 우선순위는 `docs/PROJECT_MANAGEMENT.md`의 P0~P3 기준을 따릅니다.

- **P0**: 로그인 불가, 결제 오류, 데이터 유실, 보안 문제
- **P1**: 핵심 기능 장애, 관리자 운영 차단, 심각한 모바일 사용성 문제
- **P2**: 사용성 개선, 문구·빈 상태·필터 개선
- **P3**: 장기 아이디어, 실험 기능, 디자인 디테일

## 6. Codex가 특히 조심해야 할 영역

아래 영역은 버그가 나면 영향이 크므로 작업 전후 검증을 더 엄격하게 합니다.

| 영역 | 조심해야 하는 이유 | 작업 원칙 |
|------|-------------------|-----------|
| 인증/세션 | 로그인 풀림, 권한 오류 가능 | 기존 흐름을 먼저 읽고 최소 변경 |
| 결제 | 실제 금액·크레딧 영향 | 테스트 키/운영 키 분리 확인 |
| DB/RLS | 데이터 노출·저장 실패 가능 | SQL과 적용 순서를 문서화 |
| 파일 업로드/이미지 프록시 | 보안·용량·외부 URL 문제 | 입력 검증과 실패 처리 확인 |
| AI 호출 | 비용·응답 실패 가능 | 키, 모델, 오류 메시지 확인 |
| 배포 설정 | 전체 서비스 장애 가능 | Preview/Production 분리 후 적용 |

## 7. 이전 협업 문서 처리 기준

기존 협업 문서는 삭제하지 않고 **보관용 문서**로 둡니다. 이유는 과거에 왜 그런 구조가 생겼는지 추적해야 할 때가 있기 때문입니다.

| 문서 | 현재 상태 |
|------|-----------|
| `docs/COLLABORATION_GUIDE.md` | 과거 협업 방식 기록. 신규 작업 기준 아님 |
| `docs/DB_WORKFLOW.md` | Lovable 중심 DB 절차는 보관용. 새 절차로 개정 중 |
| `docs/DEPLOY_PROCESS.md` | Lovable Publish 절차는 보관용. 새 배포 환경 확정 후 재작성 필요 |
| `docs/LOVABLE_HANDOFF.md` | Lovable 의존 정리와 제거 순서 기준 문서 |
| `docs/INDEPENDENCE_GUIDE.md` | 새 환경 전환 큰 흐름 기준 문서 |

## 8. 다음 정리 작업

Codex 중심 운영으로 완전히 바꾸려면 다음 작업을 순서대로 진행합니다.

3. 새 `.env.example` 작성

5. Lovable AI Gateway 제거 영향 조사
6. 새 배포 환경 선택 및 설정
7. 오래된 협업 문서의 세부 절차를 Codex 기준으로 재작성
