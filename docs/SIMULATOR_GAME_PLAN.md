# 대국 시뮬레이터 게임화 기획서 (2026-06-13)

> 사용자 요구: **"AI·다른 유저와 게임하듯이 대국하는 시뮬레이터. 덱 레시피를 가져와 플레이. 지금은 너무 간단한 시뮬레이터 느낌."**
> 이 문서는 현황 진단 + 외부 사례 조사 + 단계별 로드맵. 기술 스펙 원본: [SIMULATOR_SPEC.md](./SIMULATOR_SPEC.md)

## 1. 현황 진단 — 왜 "게임"이 아니라 "간단한 시뮬레이터"로 느껴지는가

### 보유 자산 (강점 — 그대로 활용)

- **순수 함수 엔진** (`applyAction(state, action) → state`) + **결정론적 RNG** — 같은 (시드, 액션열)이면 항상 같은 결과. → **PvP 동기화·리플레이가 사실상 공짜로 구현 가능한 구조.**
- 효과 DSL 인터프리터·AI 가치함수·카운터 윈도우 모델 모두 구현 완료 (`src/lib/simulator/`).
- DSL 검증 게이트 통과: 샘플 30장 중 28장(93%) 표현 가능 ([SIMULATOR_SAMPLE_CARDS.md](./SIMULATOR_SAMPLE_CARDS.md)).

### 핵심 결손 4가지 (2026-06-13 코드·DB 실측)

| # | 결손 | 근거 | 체감 |
|---|---|---|---|
| 1 | **카드 효과 데이터 0건** | DB 실측: `cards.effects` 채워진 카드 **OPTCG 740장 중 0장** (DTCG 3,203장도 0). 샘플 30장 DSL은 문서에만 있고 DB 미적재 | 모든 카드가 바닐라 스탯(코스트/파워/카운터)으로만 싸움 → 전략 깊이 없음. **"간단하다"는 느낌의 본체** |
| 2 | **게임 연출 부재** | `simulator.$id.tsx`: 버튼 리스트로 액션 선택. 드래그·공격 화살표·턴 전환 배너·KO 연출·멀리건·승패 결과 화면 없음 | 게임이 아니라 폼 입력 같음 |
| 3 | **PvP 없음** | 게임 상태가 한 클라이언트 React state에만 존재. 상대는 항상 AI | "다른 유저와 대국" 불가 |
| 4 | **덱 단절** | 메인 덱빌더(`decks`+`deck_cards`)와 시뮬 덱(`simulator_decks.recipe`)이 분리, 가져오기 경로 없음. 외부 텍스트 임포트도 없음 | 덱 레시피를 활용 못 함 |

## 3. 목표 경험 — 3가지 대국 모드

1. **AI 대국 (현행 강화)**: 효과가 실제로 발동하고, 연출이 있는 한 판.
2. **유저 PvP 친선전 (신규)**: 초대 링크 또는 로비로 1:1 실시간 대국. LFG(파트너 모집)와 연계 가능 — "온라인 대국" 카테고리.
3. **리플레이·관전 (후순위)**: 시드+액션 로그 저장으로 구현 — 엔진이 이미 지원하는 구조.

## 4. 기능 구성 상세

### A. 덱 레시피 가져오기 (즉효·저위험 — 가장 먼저)

- **내 덱 가져오기**: 덱빌더의 덱 → 시뮬 덱 변환 버튼. 양쪽 다 `card_code`+`quantity` 구조라 매핑 단순. 시뮬 미지원 카드(효과 미적재)는 바닐라로 동작함을 안내.
- **공개 덱**: `simulator_decks.is_public` 활용해 다른 유저 공개 레시피로 대국.
- **텍스트 임포트/익스포트**: `4x OP01-001` 라인 형식 (OPTCGSim 생태계 호환) — 외부에서 받은 레시피를 즉시 사용.

### B. 효과 데이터 적재 (게임 깊이의 본체)

- **1차**: SIMULATOR_SAMPLE_CARDS.md의 30장 DSL을 `cards.effects`에 적재(마이그레이션) → 프리빌트 덱 대국에서 발동 검증.
- **2차**: **프리빌트 덱 수록 카드부터** 우선 적재(즉시 체감) → 인기 리더·스타터 중심 ~100장.
- **파이프라인**: 카드 효과 텍스트 → AI가 DSL 초안 변환 → 사람 검수 → 적재. (Claude가 변환·마이그레이션 담당)

### C. 게임 UX — "게임답게" (UI 3차 톤과 같은 디자인 언어)

- **보드**: 상대 필드 상단(축소)·내 필드 하단, 부채꼴 손패, DON!!존·라이프존 시각화.
- **조작**: 카드 탭 → 확대 미리보기 → 플레이/드래그. 공격 선언 시 대상 하이라이트+화살표.
- **연출**: 턴 전환 배너, 공격 충돌·KO 페이드·라이프 데미지 셰이크, 카운터 윈도우 강조. **transform/opacity 한정, prefers-reduced-motion 존중** (기존 불변 규칙 동일).
- **게임 흐름**: 멀리건(첫 손패 1회 교체 — OPTCG 공식 룰), 승패 결과 화면(턴 수·로그 요약·다시하기·덱 변경).
- 사운드: 에셋 필요 → **보류 권장** (결정 필요 §6-4).

### D. PvP — Supabase Realtime 액션 릴레이 (lockstep)

```text
호스트가 매치 생성(시드 확정) → 게스트 입장 → 양측 같은 시드로 init
→ 내 액션을 Broadcast 채널로 전송 → 양쪽 클라이언트가 같은 applyAction 적용
→ 순수 함수 + 결정론 RNG ⇒ 두 화면이 항상 동일 상태 (서버 game loop 불필요)
```

- **테이블** `simulator_matches`: id·host_id·guest_id·seed·덱 참조·status(waiting/playing/finished)·action_log(jsonb)·winner. RLS: 참가자만 쓰기. (Claude 담당)
- **로비**: 1차 = **초대 링크**(가장 단순) → 2차 = 공개 매치 목록 → 3차 = LFG quick_match 연계.
- **끊김 복구**: action_log가 DB에 누적되므로 재접속 시 리플레이로 상태 복원.
- **치팅 방지 수위**: 친선전은 클라이언트 신뢰 모델로 충분(상대 손패는 화면에서 숨김). 랭크전 도입 시 Edge Function 권위 서버(서버가 액션 검증·은닉 정보 관리)로 승격 — **후순위, 지금 설계만 호환되게**.

## 5. 로드맵 (단계 → 검증 방법)

| Phase | 내용 | 검증 목표 |
|---|---|---|
| **1. 덱+효과** | A(가져오기 3종) + B 1차(30장 적재) | "덱빌더의 내 덱으로 AI 대국, 등장 시 효과·블로커·카운터가 실제 발동" |
| **2. 게임 UX** | C 전체 (보드 개편·연출·멀리건·결과 화면) | 폰에서 한 판 완주 후 사용자 평가 "게임 같다" |
| **3. PvP 친선** | D (테이블+RLS+Broadcast+초대 링크) | 브라우저 2개로 실제 1:1 대국 완주, 새로고침 복구 동작 |
| **4. 확장** | 공개 로비·LFG 연계·리플레이·효과 ~100장·전적 연동 | 항목별 별도 정의 |

**분담**: DB·RLS·Realtime 설계·DSL 변환 = Claude / 보드 UI·연출 = Antigravity·Codex / 검증 = Claude+사용자.

## 6. 결정 필요 (사용자 확인 후 착수)

1. **Phase 순서 승인** — 권장: 1(깊이) → 2(겉) → 3(PvP). PvP를 먼저 원하면 3을 앞당길 수 있으나 "바닐라 카드끼리 PvP"가 됨.
2. **UI 3차(톤 통일)와의 순서** — 권장: 진행 중인 3차를 먼저 완료(3차의 시뮬레이터 대국 화면은 보수적 적용이라 충돌 적음), Phase 2에서 보드를 새로 만들 때 같은 game 토큰 사용.
3. **PvP 결과를 공식 전적(matches)에 기록할지** — 기록 시 시즌 승률에 영향.
4. **사운드 도입 여부** — 에셋 소싱 필요, 보류 권장.
