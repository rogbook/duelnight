# 하네스 설계 원칙 (Agent Harness)

> 코딩 AI 에이전트 하네스 설계 강의(2026, 사내 자료)의 핵심 정리.
> 작업 자세의 단일 기준은 [AGENTS.md](../AGENTS.md) §2-1 + [CLAUDE.md](../CLAUDE.md) §1·§1-1.
> 이 문서는 "왜 그렇게 일하는가"의 배경 — 필요할 때만 펼쳐 읽는다(지식 맵 점진 공개).

## 1. 하네스란

모델(LLM) 자체가 아니라, **모델을 감싸 실제 일을 시키는 바깥 구조 전체**.
컨텍스트·지식, 도구·환경, 루프·검증, 메모리·관측이 여기 속한다.

> **에이전트 성능 = 모델 × 하네스.** 같은 모델이라도 하네스가 부실하면 20분 만에 무너지고(솔로 $9), 잘 설계하면 6시간·16기능을 정상 수행한다(풀 하네스 $200). — 강의 PART 5

## 2. 세 축 (설계 영역)

### AXIS 1 — 지식 맵

- 루트 문서(`AGENTS.md`/`CLAUDE.md`)는 **모든 상세를 담는 곳이 아니라 "안내판"** — 무엇을 먼저 읽고 무엇을 피할지 알려주는 진입점.
- 상세 기준 문서는 `docs/` 하위에 분리(설계·수용기준·실행계획·참고자료).
- 에이전트는 처음부터 전부 읽지 않는다. **필요한 문서만 점진적으로 열어 컨텍스트 예산을 아낀다.**
- 우리 적용: `CLAUDE.md`(품질·스킬), `AGENTS.md`(역할·절차), `docs/SIMULATOR_*`(기능 스펙).

### AXIS 2 — 도구·환경 설계

- 도구는 self-contained(도구 간 overlap 최소화, 명확한 입출력 계약). **에러 메시지 = 수정 지침**이 되도록.
- 권한은 승인 기반 루프 + 정책/분류로 자동화. 트레이드오프: Sandbox(안전·유지보수↑) / Skip-perms(빠름·위험↑) / 수동 승인(안전·피로↑).
- 우리 적용: bun build 게이트, DB 변경은 Claude만, 보안 코드 Claude 1회 검토.

### AXIS 3 — Eval 기반 개선

- **"느낌으로 개선하지 말라"** — 결과물뿐 아니라 process quality(어떤 스킬·도구로 그 결과가 나왔는지)까지 측정.
- Evaluation Harness 구성요소: Task(평가 작업 정의) · Trial(여러 번 실행으로 분산 감소) · Grader(자동/수동 채점) · Transcript(실행 기록) · Outcome(집계·비교).
- 실무 첫걸음: 거창한 벤치마크보다 **실제 이슈 10~20개를 재현하는 mini regression set**.
- 자기평가의 함정: "자기 작업을 채점하면 형편없어도 자신 있게 칭찬한다" → **독립 evaluator(교차 검증)** 필요.
- 우리 적용: `scripts/verify-simulator-phase1.ts`(6시나리오 결정론 검증), `scripts/build-effect-migration.ts`(zod 게이트 후 적재).

## 3. 좋은 하네스의 4원칙 (PART 2)

1. **가장 단순한 구성부터** — 처음부터 복잡한 프레임워크 금지. 작은 성공 사례 후 필요한 복잡도만 추가.
2. **길이보다 high-signal** — 컨텍스트는 길다고 좋은 게 아니다. 목표에 직접 필요한 정보만 남겨야 모델이 집중한다.
3. **검증을 하네스에 내장** — 테스트·규칙·스크린샷·grader를 루프 안에 넣는다. 검증이 밖에 있으면 개선이 느려지고 회귀를 놓친다.
4. **가정을 정기적으로 재검토** — 각 구성요소는 모델 한계를 가정한 우회. 모델이 좋아지면 불필요해질 수 있으니 주기적으로 재평가.

## 4. 권장 아키텍처 (PART 2)

```
Input(User/Spec) ┐
                 ├→ Agent Harness Controller(루프·라우팅·정책) → Tool Layer(Shell/FS/Browser/MCP, Sandbox/Runner)
Context(지식 맵) ┘                                                  ↓
                          ┌──────────── feedback loop ──────────── Observe(Traces·Logs·Diffs) + Verify(Lint·Tests·Evals)
```

4블록 분리(지식 맵·컨트롤러·도구/환경·관측/검증)로 "교체 가능 + 안전 경계 명확화".

## 5. 멀티 에이전트 패턴 (우리 체제의 이론적 근거)

- **2중(교차) 검증 루프**: 같은 모델이 만든 것을 같은 모델이 검증하지 않는다. 만든 주체 ≠ 검증 주체로 blind spot 제거. → 우리: 부 개발자(Antigravity/Codex) 구현 → **Claude 검수·병합**.
- **Ouroboros(맞물림) 루프**: 한 사이클의 출력이 다음 사이클의 입력. 병렬이 아닌 **맞물림** — 품질이 수렴할 때까지 순환.
- **정책 파일·Skills 공용화**: 운영 지식을 특정 도구에 묶지 말고 repo 안 canonical artifact로 공유. 진짜 지켜야 할 규칙은 "문서"가 아니라 "코드(Hooks·CI Gates·Linters·Sandbox)"로 강제.
- **이슈→PR 파이프라인**: Triage&Spec(사람+CC) → Parallel Exec(부 개발자) → Eval&Verify(자동) → Review&Merge(사람+CC).

## 6. 리스크 — 하네스는 새로운 공격면 (PART 5)

| 리스크          | 내용                                                   | 대응                          |
| --------------- | ------------------------------------------------------ | ----------------------------- |
| 승인 피로       | 반복 승인 후 검토 소홀("그냥 클릭")                     | Auto mode 분류기·허용목록·샌드박스, **무거운 명령 남발 금지** |
| 도구 오염(MCP)  | 악성 MCP가 도구 설명에 숨긴 지침으로 키·설정 탈취       | 버전 고정·변경 알림·신뢰 서버만 사용 |
| 코드 드리프트   | 시간이 지나며 아키텍처 이탈, 복붙 증가·리팩터링 감소    | 커스텀 린터·hooks·**최소 diff·기존 패턴 재사용** |
