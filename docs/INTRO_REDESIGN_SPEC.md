## 1. 배경 / 문제 정의

- 직전 커밋(`452f9b9`)에서 intro.tsx를 게이밍 히어로 + 리더보드 카드 중심으로 개편했으나, 운영 기획 방향과 맞지 않음.
- 기획이 원하는 그림:
  1. **최상단**: 각 부스터팩 신규 발매 일정을 **스크롤형 배너**로 노출
  2. **하단**: 프로필 화면 + 전적을 **게이미피케이션 형태**로 노출 (dak.gg 성적표 느낌)
- dak.gg 성적표(`/er/season-report/{닉네임}`)의 핵심 UX를 개인 대시보드/메인에 이식하고 싶음.

### 핵심 제약 — "비로그인" 화면이라는 점

인트로는 로그인하지 않은 방문자가 보는 화면이다. 따라서 "내 전적/내 프로필"을 직접 표시할 수 없다.
→ **해결책**: 리더보드 **TOP 플레이어의 시즌 성적표 카드를 쇼케이스**로 노출한다.

- 소셜 프루프(실데이터 기반 신뢰) + 게이미피케이션(성적표가 멋있어 보임) → **회원가입 전환** 유도.
- 방문자에게 "가입하면 나도 이런 성적표가 생긴다"를 보여주는 구조.

## 2. 정보 구조 (IA) — 위에서 아래로

```
┌─────────────────────────────────────────────┐
│ 0. 상단바 (로고 / 언어 / 로그인·무료시작)        │  ← 기존 유지
├─────────────────────────────────────────────┤
│ 1. 신규 발매 스크롤 배너 (ReleaseTicker)        │  ← 기존 컴포넌트 재사용·강화
├─────────────────────────────────────────────┤
│ 2. 히어로 (서비스 한 줄 + 게임 탭 + CTA)         │  ← 축소·정리
├─────────────────────────────────────────────┤
│ 3. ★ 시즌 성적표 쇼케이스 (NEW, 메인 볼거리)      │  ← dak.gg 성적표 이식 (핵심)
│    - TOP 플레이어 프로필 + 티어/랭크 헤더         │
│    - 시즌 요약 스탯 카드 (승률/연승/선후공/판수)   │
│    - 모스트 덱 Top3                              │
│    - 모스트 매치업(상대 메타) Top                 │
├─────────────────────────────────────────────┤
│ 4. 리더보드 미니 (TOP 3 다른 플레이어)           │  ← 기존 GamingHero 카드 일부 흡수
├─────────────────────────────────────────────┤
│ 5. 기능 소개 (Features) / 가격 (Pricing)        │  ← 기존 유지, 위치만 하단으로
├─────────────────────────────────────────────┤
│ 6. 하단 CTA + 푸터                              │  ← 기존 유지
└─────────────────────────────────────────────┘
```

가장 큰 변화: **3번 "시즌 성적표 쇼케이스"가 신규 핵심 섹션**. 기존 GamingHero의 TOP3 카드 나열은 4번 "리더보드 미니"로 축소.

## 3. 섹션별 상세 설계

### 1) 신규 발매 스크롤 배너 (`ReleaseTicker` 강화)

- **현행 유지 + 소폭 강화**. 이미 `events(kind='release')`에서 가져와 무한 가로 스크롤 구현됨.
- 강화 포인트:
  - 게임별 색 테마(`GAME_THEME`) 유지.
  - D-day 뱃지 추가: `early_release_at`/`starts_at` 기준 `D-7`, `오늘 발매` 등. (운영 기획 "발매 일정 배너" 요구의 핵심)
  - 호버 시 일시정지(이미 구현됨) 유지.
- 데이터: `supabase.from("events").eq("kind","release")` — 변경 없음.

### 2) 히어로 (축소)

- 한 줄 타이틀 + 서비스 설명 + 게임 탭(`optcg/ptcg/dtcg`) + 메인 CTA("무료로 시작").
- 게임 탭 선택값은 **3번 성적표 쇼케이스 / 4번 리더보드와 연동**(선택 게임의 TOP 플레이어를 표시).
- 기존 GamingHero의 거대한 배경 글로우는 유지하되 높이 축소.

### 3) ★ 시즌 성적표 쇼케이스 (NEW — dak.gg 이식 핵심)

선택된 게임의 **리더보드 1위 플레이어**를 대상으로 성적표를 렌더링.

#### 3-a. 성적표 헤더 (프로필 + 게이미피케이션)

```
[아바타]  닉네임 (@username)              [티어 뱃지: 예) 챌린저]
          ONE PIECE · 이번 시즌            상위 X% · 1234 RP
```

- **티어 뱃지(게이미피케이션)**: `rating` 값을 구간 매핑하여 티어 산출 (아래 §4 티어 테이블).
- **상위 백분위**: 리더보드 순위 / 전체 인원으로 근사. (1위이므로 "상위 0.x%" 또는 "RANK #1" 표기)
- "이번 시즌" = 최근 90일 (대시보드 `SEASON_START` 기준과 동일하게 통일).

#### 3-b. 시즌 요약 스탯 카드 (가로 스크롤 / 데스크탑은 grid)

dak.gg 성적표 상단 요약과 동일한 결.

| 카드       | 값                | 부가설명               |
| ---------- | ----------------- | ---------------------- |
| 종합 승률  | `overall.winRate` | `n승 m패 · 총 k판`     |
| 현재 연승  | `streak.current`  | 🔥 / 최고연승·최저연패 |
| 선공 승률  | `first.winRate`   | 선공 판수              |
| 후공 승률  | `second.winRate`  | 후공 판수              |
| 사용 덱 수 | `byDeck.length`   | 매치업 수              |

→ 계산은 `src/lib/match-stats.ts`의 `computeStats()` / `computeStreak()` **재사용**.
→ 데이터는 `get_user_recent_matches(game, user_id, limit)` 공개 RPC로 해당 플레이어 전적 fetch.

#### 3-c. ★ 모스트 덱 Top3 (사용자 선택 게이미피케이션 1순위)

dak.gg "모스트 캐릭터"에 대응.

```
┌──────────────┐ ┌──────────────┐ ┌──────────────┐
│ #1 적 루피     │ │ #2 흰수염     │ │ #3 에넬       │
│ 승률 68%      │ │ 승률 55%      │ │ 승률 50%      │
│ ▓▓▓▓▓▓░░ 68% │ │ ▓▓▓▓▓░░░ 55% │ │ ▓▓▓▓░░░░ 50% │
│ 17승 8패·25판 │ │ ...           │ │ ...           │
│ 선공68 후공... │ │               │ │               │
└──────────────┘ └──────────────┘ └──────────────┘
```

- 데이터: `computeStats().byDeck` (사용 판수 기준 정렬, 상위 3).
- 표시: 덱명, 종합 승률(큰 글씨), 승률 막대바, 전적(W-L·판수), 선후공 승률.
- Wilson 하한(`wilsonLow`)으로 표본 적은 덱 과대평가 방지 — 정렬 보조 지표로 활용 가능.

#### 3-d. ★ 모스트 매치업 / 상대 메타 (사용자 선택 게이미피케이션 1순위)

dak.gg "상대 / 카운터" 느낌.

```
상대 메타 (가장 자주 만난 상대 덱)
적 루피  vs  카이도     승률 70%   ▓▓▓▓▓▓▓░ 10판
흰수염   vs  에넬       승률 40%   ▓▓▓░░░░░ 8판
...
```

- 데이터: `computeStats().matchups` (판수순 상위 N, 기본 6) 또는 `opponents`(상대 빈도).
- 표시: 내 덱 vs 상대 덱, 승률, 빈도 막대, 전적.

> 모바일은 기존 `match-stat-cards.tsx`의 가로 스크롤 카드 패턴(`MobileDeckCards`, `MobileMatchupCards`)을 **그대로 차용**하면 일관성↑·작업량↓.

### 4) 리더보드 미니 (TOP3)

- 기존 GamingHero의 TOP3 카드 컴포넌트를 분리/축소하여 이 위치에 배치.
- "전체 랭킹 보기" → `/leaderboard` 링크 유지.

### 5) 기능 소개 / 가격

- 기존 `Feature`, `PriceCard`, `MobileIntro` 아코디언 **그대로 유지**, 위치만 성적표 쇼케이스 아래로 이동.

### 6) 하단 CTA + 푸터

- 기존 유지.

## 4. 게이미피케이션: 티어 테이블 (rating → 티어)

dak.gg식 등급 시각화를 위해 `rating` 정수값을 구간 매핑. (초기값, 운영 데이터 쌓이면 재조정)

| 티어                | rating 구간 | 색상 키         |
| ------------------- | ----------- | --------------- |
| 챌린저 (Challenger) | 1600+       | amber/gold glow |
| 다이아 (Diamond)    | 1400–1599   | sky/cyan        |
| 플래티넘 (Platinum) | 1250–1399   | teal            |
| 골드 (Gold)         | 1100–1249   | yellow          |
| 실버 (Silver)       | 950–1099    | slate           |
| 브론즈 (Bronze)     | ~949        | orange/brown    |

- 기본 rating 시작값(예: ELO 1000~1200) 확인 후 구간 보정 필요 → **TODO: `user_ratings` 초기값/분포 확인**.
- 백분위는 `get_leaderboard`로 받은 순위 / 총 인원 근사. 별도 RPC 추가 시 정확도↑ (선택).

## 5. 데이터 소스 매핑 (전부 기존 공개 자원으로 충당 가능)

| 섹션                | 소스                                                    | 비고              |
| ------------------- | ------------------------------------------------------- | ----------------- |
| 발매 배너           | `events` (kind=release)                                 | 기존              |
| 성적표 대상 선정    | `get_leaderboard(game, limit=1, min_total=1)`           | TOP1 user_id 획득 |
| 성적표 전적         | `get_user_recent_matches(game, user_id, limit)`         | **공개 RPC**      |
| 요약/덱/매치업 통계 | `src/lib/match-stats.ts` `computeStats`/`computeStreak` | 클라 계산, 재사용 |
| 티어/백분위         | `get_leaderboard` rating + 순위                         | 클라 매핑         |
| 리더보드 미니       | `get_leaderboard(game, limit=3)`                        | 기존              |

> **신규 마이그레이션 불필요(권장)**. `get_user_recent_matches`의 `limit`을 시즌 통계에 충분한 값(예: 200)으로 호출.
> 만약 RPC가 시즌(90일) 필터를 못 한다면, 클라에서 `played_at >= SEASON_START`로 필터. (현재 반환에 `played_at` 포함됨 → 가능)

## 11. 로그인 후 홈/대시보드 재설계 (`index.tsx`)

### 11-0. 핵심 차이 — 인트로 vs 홈

|             | 인트로 (비로그인)       | 홈/대시보드 (로그인)     |
| ----------- | ----------------------- | ------------------------ |
| 성적표 대상 | TOP 플레이어 (쇼케이스) | **본인(나)**             |
| 목적        | 가입 전환               | 내 시즌 현황 한눈에      |
| 데이터 권한 | 공개 RPC만              | 내 데이터 직접 조회 가능 |

> **결정적 이점**: 홈은 §3에서 설계한 **시즌 성적표 쇼케이스 컴포넌트를 "나"에게 그대로 재사용**한다.
> 같은 `SeasonReport` 컴포넌트에 `userId`만 바꿔 넣으면 됨 → 인트로/홈이 동일 UI를 공유.

### 11-1. 현행 홈의 문제

현재 `index.tsx`는:

- 단순 4칸 스탯(`승률/카드/덱/레이팅`) + 8개 바로가기 그리드.
- 정보 밀도는 있으나 **"게이미피케이션·성적표 느낌"이 전혀 없음**. 운영 기획 방향과 불일치.

### 11-2. 새 홈 정보 구조 (위→아래)

```
┌─────────────────────────────────────────────┐
│ 0. PageHeader + GameFilter                    │  ← 기존 유지 (게임 필터 활용)
├─────────────────────────────────────────────┤
│ 1. 신규 발매 스크롤 배너 (ReleaseTicker 재사용)  │  ← 인트로와 공유, 기획 "최상단 배너"
├─────────────────────────────────────────────┤
│ 2. ★ 내 시즌 성적표 (My Season Report)          │  ← §3 컴포넌트 재사용, userId=나
│    - 내 프로필 + 티어 뱃지 + 상위 %             │
│    - 시즌 요약(승률/연승/선후공/판수)            │
│    - 내 모스트 덱 Top3                          │
│    - 내 상대 메타(매치업)                        │
│    - [내 전적 전체 보기] → /matches             │
├─────────────────────────────────────────────┤
│ 3. 핵심 KPI 미니 (카드 수 / 덱 수 / 다음 일정)    │  ← 기존 스탯 중 비-전적 항목 압축 유지
├─────────────────────────────────────────────┤
│ 4. 바로가기 그리드 (Quick Links)                │  ← 기존 8개 유지, 위치만 하단
└─────────────────────────────────────────────┘
```

### 11-3. 섹션별 처리

- **1. 발매 배너**: 인트로와 동일 `ReleaseTicker` 컴포넌트 공유 (분리하여 양쪽에서 import).
- **2. 내 시즌 성적표 (핵심 신규)**:
  - 데이터: 인트로의 공개 RPC 대신 **내 데이터 직접 조회** 가능.
    - 전적: `supabase.from("matches").eq("user_id", me).gte("played_at", SEASON_START)` (기존 대시보드 쿼리 재활용) → `computeStats`/`computeStreak`.
    - 레이팅/티어: 기존 `user_ratings` 쿼리(현 대시보드 `rankData`) 재사용.
  - UI: §3-a~3-d 그대로. "쇼케이스"가 아니라 "내 현황"이므로 카피만 1인칭("내 승률", "내 모스트 덱")으로.
  - **빈 상태**: 전적 없으면 "첫 전적을 기록해보세요" → `/matches` CTA (가입 유도 대신 사용 유도).
- **3. KPI 미니**: 기존 4칸 중 전적/레이팅은 성적표로 흡수되므로, **보유 카드 수 / 덱 수 / 다가오는 일정** 정도만 슬림하게 유지.
- **4. 바로가기**: 기존 `localizedShortcuts` 8개 그대로, 맨 아래로.
