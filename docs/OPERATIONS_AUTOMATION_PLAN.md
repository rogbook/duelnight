# DuelNight 운영 자동화 계획 (Operations Automation Roadmap)

작성일: 2026-06-14 | 작성: Claude | 대상: 멀티 AI 협업 체제 (Claude · Codex · Antigravity)

> 전제: 인프라는 서버리스(Cloudflare Workers + Supabase + Gemini)로 확정([OPERATIONS_PLAN.md](./OPERATIONS_PLAN.md)).
> 이 문서는 **"오픈 후 1인 운영을 버티게 하는 자동화"** 의 단일 계획서다. 무엇을 / 왜 / 어떻게 / 누가 / 비용을 한곳에 모은다.
> 구현 착수 전 사용자 승인을 받는다. 항목별로 독립적이라 우선순위대로 부분 착수 가능.

---

## 0. 핵심 원칙

- **1인+AI 운영이 전제.** 사람이 매일 대시보드를 들여다보지 않아도 "문제가 생기면 먼저 알림이 온다"가 목표.
- **서버리스 유지.** 별도 모니터링 서버·VPS 신설 금지. 전부 GitHub Actions(cron) · Supabase(pg_cron) · 외부 무료 SaaS 무료 구간으로 해결.
- **비용 0원에서 시작.** 트래픽이 커지기 전엔 전 항목 무료 구간 안에서 운영.
- **푸시형 알림 한 채널로 집중.** 알림은 분산시키지 않고 **단일 채널(이메일 또는 Slack/디스코드 웹훅)** 로 모은다(피로도·누락 방지).

## 1. 현재 상태 진단 (2026-06-14)

| 영역             | 현재                                                              | 자동화 수준    |
| ---------------- | ----------------------------------------------------------------- | -------------- |
| 배포             | Cloudflare Workers, `main` push → 자동 배포                       | ✅ 자동        |
| CI 검증          | `guard-lovable-files.yml` (PR 보호 파일 가드)만 존재              | ⚠️ 부분 (검증 없음) |
| DB 백업          | `scripts/backup-db.ts` (PII 마스킹 스냅샷) — **수동 실행**        | ❌ 수동        |
| 에러 추적        | 없음 (콘솔/CF 대시보드 수동 확인)                                 | ❌ 없음        |
| 업타임 감시      | 없음                                                              | ❌ 없음        |
| 비용 감시        | 없음 (Gemini·Supabase·CF 한도 도달 시 사후 인지)                  | ❌ 없음        |
| 보안 점검        | `get_advisors` 수동, 의존성 자동 업데이트 없음                    | ❌ 수동        |
| DB 정리/집계     | cron 작업 없음 (만료 세션·임시데이터·통계 집계 수동)             | ❌ 없음        |
| 인시던트 대응    | 런북 문서 없음                                                    | ❌ 없음        |

**결론:** 배포만 자동화되어 있고, **"운영 중 문제 감지·대응·백업"이 전부 비어 있다.** 외부 테스터/정식 오픈 전에 최소 1~2단계는 갖춰야 한다.

## 2. 자동화 대상 전체 맵

### A. 관측성 (Observability) — "무슨 일이 일어나는지 안다"

| 항목             | 수단                                                          | 비용        | 담당   |
| ---------------- | ------------------------------------------------------------- | ----------- | ------ |
| 에러 추적        | **Sentry** (프론트 + Worker) 무료 5k err/월                   | 0원         | Claude |
| 업타임/헬스체크  | `/health` 엔드포인트 + **UptimeRobot**(무료 50모니터·5분)     | 0원         | Claude |
| 핵심 경로 모니터 | 합성 점검(로그인·카드DB·AI코치 e2e)를 GitHub Actions cron으로 | 0원         | Codex  |
| 트래픽/성능      | Cloudflare Analytics(기본 제공) + Web Analytics 토큰          | 0원         | Claude |
| 구조적 로그      | Worker `console` → CF Logs / Logpush(성장 시)                 | 0원~        | Claude |

### B. 신뢰성 (Reliability) — "데이터를 잃지 않는다"

| 항목                 | 수단                                                                       | 비용         | 담당   |
| -------------------- | -------------------------------------------------------------------------- | ------------ | ------ |
| 정기 DB 백업         | `backup-db.ts`를 **GitHub Actions cron(일 1회)** 으로 → 아티팩트/스토리지 보관 | 0원          | Claude |
| Supabase PITR        | 정식 오픈 전 **Supabase Pro($25/월)** 전환 시 자동 백업·시점복구 활성       | $25/월(오픈시)| 사용자+Claude |
| 백업 복구 리허설     | 분기 1회 복구 테스트 절차(런북)                                             | 0원          | Claude |
| 배포 롤백            | CF 배포 버전 롤백 절차 문서화 + (선택)배포 후 스모크 실패 시 자동 롤백 액션 | 0원          | Claude |

### C. 품질 게이트 (CI) — "깨진 코드가 배포되지 않는다"

| 항목             | 수단                                                              | 비용  | 담당   |
| ---------------- | ----------------------------------------------------------------- | ----- | ------ |
| PR 빌드/타입검사 | GitHub Actions: `bun install` → `bun run build`(tsc 포함) → `lint` | 0원   | Codex  |
| 자동 테스트      | 기존 `bun test`(sync-db/migrate-images) PR마다 실행                | 0원   | Codex  |
| 배포 후 스모크   | 배포 완료 후 `/health` + 핵심 SSR 라우트 200 확인                  | 0원   | Codex  |

### D. 보안 자동화 — "취약점이 쌓이지 않는다"

| 항목                 | 수단                                                         | 비용  | 담당   |
| -------------------- | ------------------------------------------------------------ | ----- | ------ |
| RLS/보안 어드바이저  | `get_advisors(security)`를 **주 1회 cron**, 신규 경고 시 알림 | 0원   | Claude |
| 의존성 업데이트      | **Dependabot**(주간) — 보안 패치 자동 PR                     | 0원   | Claude |
| 시크릿 스캔          | GitHub Secret Scanning + push 전 가드(이미 .env 차단)        | 0원   | Claude |
| 코드 스캔            | CodeQL(선택, 무료) — 성장기                                  | 0원   | Claude |

### E. 비용·사용량 감시 — "요금 폭탄을 막는다"

| 항목             | 수단                                                                  | 비용  | 담당   |
| ---------------- | --------------------------------------------------------------------- | ----- | ------ |
| AI(Gemini) 사용량 | `gemini-proxy`에 호출/토큰 로깅 테이블 → 일 집계 → 임계치 알림         | 0원   | Claude |
| Supabase 한도    | DB 용량·MAU·Egress 주간 리포트(API 조회) → 80% 도달 시 알림           | 0원   | Claude |
| Cloudflare 한도  | 요청 수 추세 점검(무료 10만/일)                                        | 0원   | Claude |

### F. 정기 운영 작업 (Scheduled Jobs) — "유지보수를 잊지 않는다"

| 항목                  | 수단                                       | 비용  | 담당   |
| --------------------- | ------------------------------------------ | ----- | ------ |
| 만료/임시 데이터 정리 | **Supabase pg_cron**(만료 세션·임시 업로드) | 0원   | Claude |

| 알림/공지 발송 잡     | (필요 시) Edge Function + cron              | 0원   | Claude |

### G. 운영 가시화 — "한눈에 본다"

| 항목            | 수단                                              | 비용  | 담당        |
| --------------- | ------------------------------------------------- | ----- | ----------- |

## 3. 우선순위 로드맵 (오픈 일정 기준)

> 단계는 독립적. **1단계는 외부 테스터 확대 전, 2단계는 정식 오픈 전** 완료를 권장.

### 🟢 1단계 — 안전망 (외부 테스터 확대 전 / 즉시 착수 후보)

목표: **장애·데이터 손실을 "사후"가 아니라 "즉시" 알게 된다.**

1. **CI 품질 게이트** — PR마다 build+typecheck+lint+test (가장 비용 낮고 효과 큼)
2. **`/health` 엔드포인트 + UptimeRobot** — 다운 즉시 알림
3. **자동 DB 백업** — `backup-db.ts`를 GitHub Actions 일 1회 cron화 + 보존
4. **Sentry 에러 추적** — 프론트/Worker 런타임 에러 수집·알림
5. **알림 채널 단일화** — 위 4개의 알림을 한 채널(이메일/웹훅)로

→ 산출물: 워크플로 4종(`ci.yml`, `backup.yml`, `synthetic.yml`, dependabot) + `/health` 라우트 + Sentry 초기화 + 런북 v1

### 🟡 2단계 — 운영 성숙 (정식 오픈 전)

목표: **비용·보안·데이터 위생을 자동으로 유지.**

6. **주간 보안 어드바이저 cron** + Dependabot 활성
7. **AI/Supabase/CF 비용·한도 감시 리포트**(주간) + 임계치 알림
8. **Supabase Pro 전환** — PITR/자동백업 (사용자 결제 결정 필요)
9. **pg_cron 정리 잡** — 만료 데이터·임시 업로드 정리
10. **배포 후 스모크 + 롤백 절차** 문서화

### 🔵 3단계 — 규모화 (트래픽/매출 발생 후)

11. 합성 e2e 모니터(핵심 사용자 여정) 상시화
12. 리더보드/통계 야간 집계 배치
13. 관리자 운영 상태 페이지(헬스·백업·비용 요약 탭)
14. (선택) CodeQL, Logpush, 상태 페이지 공개

## 4. 알림 설계 (단일 채널 원칙)

- **채널 후보**: ① 이메일(가장 단순) ② Slack/디스코드 incoming webhook(이미 Slack MCP 연동 있음).
- **심각도 2단계로만** 운영 (과알림 방지):
  - 🔴 **즉시(P1)**: 사이트 다운, DB 백업 실패, 신규 보안 경고(critical), 비용 한도 90%↑
  - 🟡 **다이제스트(주간)**: 의존성 PR 요약, 사용량/비용 추세, advisor 경고(low/med)
- 모든 알림은 **무엇이·언제·조치링크** 3요소를 포함.

## 6. 다음 액션 (사용자 승인 대기)

**즉시 착수 추천(가성비순):** ① CI 품질 게이트 → ② `/health`+UptimeRobot → ③ 자동 DB 백업 cron.
이 3개는 **비용 0원, 코드 변경 최소, 위험 거의 없음**이며 운영 안정성 체감이 가장 크다.

> 확인 필요 사항:
> 1. 알림 채널을 **이메일**로 할지 **Slack/디스코드 웹훅**으로 할지
> 2. **Supabase Pro($25/월) 전환 시점** (PITR 자동백업 — 정식 오픈 전 권장)
> 3. Sentry 등 외부 SaaS 계정 생성 주체(사용자 계정 vs 신규)
>
> 승인 시 Claude가 1단계부터 워크플로/스크립트를 구현하고 항목별로 검증·보고한다.

## 7. 역할 분담 (AGENTS.md §1 준수)

- **Claude(리드)**: 인프라·DB·보안·백업·비용 감시 설계 및 구현, cron/pg_cron, Sentry/Workers 연동, 런북.
- **Codex**: CI 워크플로(build/test/lint), 합성·스모크 점검 스크립트, 브라우저 검증.
- **Antigravity**: 관리자 운영 콘솔/상태 페이지 UI.

> DB 스키마·RLS·키·pg_cron 변경은 **Claude만** 실행한다(AGENTS.md §1 인터페이스 규칙).
